Self-Evolving Agent 是下一代 Agent 吗？

它很可能是下一代 agent 的核心能力之一，但不是全部。下一代 agent 还需要手、眼、记忆、验证器、工具系统、协作机制和安全边界。

核心能力：学习与自我改进 落地路径：Eval-first + Computer-use + Workflow + Memory 长期形态：可控的自我进化系统
Thesis

Self-evolving agent 是下一代 agent 的“学习能力”，但下一代 agent 不是单靠 self-evolution 就能成立。

它解决的是当前 agent 最大的短板：脆、不稳定、经验不能复用、失败后不会真正变好。

1. 业内还有哪些方向

这些方向不是互斥的。真正有用的下一代 agent，大概率是多条线合流之后的系统，而不是单点技术突破。

1

Computer-use / GUI Agent

能真正操作电脑、浏览器、手机、App、文件系统的 agent。解决 agent 的“手和眼”。

2. 一张图

下一代 agent 不是一个单独模块，而是一组能力叠起来的系统。

下一代 Agent
├─ 会操作真实环境：computer-use / browser / app / OS
├─ 会做长任务：planning / recovery / verification
├─ 会协作：multi-agent / population / specialization
├─ 会积累经验：memory / skill / workflow memory
├─ 会自我改进：self-evolving / harness evolution / code evolution
├─ 可验证：eval-first / verifier / regression / safety
├─ 可工程化：agentic workflow / tool protocol / sandbox / audit
└─ 可个性化与垂直化：personal agent / domain agent

3. 时间判断

短期 1-2 年

最先落地的不是完全自主 self-evolving agent，而是 eval-first + computer-use + workflow + memory 的组合。

中期 2-4 年

self-evolving 会变成成熟 agent 的标配能力：每完成一批任务，就能更新技能库、失败模式、workflow、tool wrapper 和 evaluator。

更长期

DGM 这种 code-level self-improvement 可能成为真正的下一代 agent engine，但会受到 sandbox、审计、人类审批和法律安全边界强约束。

4. MVP 路线

如果现在要做，不建议第一天就做“自动改代码”或“完全自主进化”。更稳的路线是分两步：第一步先做一个能稳定完成真实任务、能被验证的 agent；第二步再让它从任务轨迹里沉淀经验，形成轻量 self-evolution。

MVP Principle

第一步先证明 agent 能可靠干活；第二步再证明 agent 能越用越好。

1

第一步 MVP：可验证执行型 Agent

目标不是 self-evolve，而是做一个在单一垂直场景里能真实执行、可观测、可验收、可回滚的 agent。它要能操作真实环境，但所有关键动作都被 workflow 和 verifier 约束。

Eval-firstComputer-useWorkflowTrace Log
2

第二步 MVP：轻量 Self-Evolving Agent

在第一步稳定后，开放低风险的自进化对象：memory、failure pattern、skill card、workflow checklist、tool policy。agent 可以提出更新，但高风险更新先 human-gated。

MemorySkillWorkflow UpdateHuman-gated

第一步 MVP 具体做什么

模块要做什么验收标准
垂直任务只选一个高频、边界清楚的场景，例如研究摘要、代码仓库问答、浏览器填表、CRM 更新。任务输入和完成状态可定义，不追求通用。
Workflow把任务拆成固定步骤：接收任务、检索上下文、执行、阶段验证、失败恢复、人工确认、最终提交。每一步有进入条件、退出条件和失败分支。
Computer-use / Tools接入浏览器、文件、API、IDE 或 SaaS 工具，但先限制工具集合和权限。能完成真实环境操作，关键动作有日志。
Eval / Verifier先设计验收器：状态检查、测试、格式检查、引用检查、安全检查、成本检查。不是 agent 说完成，而是 verifier 判定完成。
Trace Log记录每次任务的输入、计划、动作、工具调用、观察、错误、最终输出、验收结果。失败后能复盘到具体步骤和工具调用。
Human Approval发送邮件、提交订单、删除文件、付款、外部发布等动作必须停下来确认。高风险动作不会自动越权执行。

第一步 MVP 的成功标准：不是“agent 很聪明”，而是它在一个具体场景里能稳定完成任务，并且每次成功或失败都能被验证、被复盘、被回滚。

第二步 MVP 具体做什么

可进化对象自动生成什么如何防止越进化越差
Memory从成功和失败轨迹中生成用户偏好、工具坑点、任务 SOP、失败模式。保留 raw trace；每条 memory 带 source_run_id、适用范围、置信度和过期条件。
Skill Card把重复成功流程沉淀成 skill：intent、preconditions、steps、validation、failure modes。skill 必须有 probe task 或 validation checklist；低成功率自动降权。
Workflow Checklist根据失败归因给 workflow 加步骤、加 recovery branch、加 human approval point。新 workflow 先在 held-out / regression 任务上验证，不能只修当前案例。
Tool Policy生成工具使用提示、pre-check、post-check、风险等级、失败恢复策略。权限、发送、删除、付款等高风险策略必须人工确认。
Evaluator Notes总结 verifier 漏判、误判、需要新增的检查项。evaluator 不能自己改 gold labels；新 verifier 要过校准集。
Failure Taxonomy把失败归类为上下文缺失、工具错误、流程错误、验证缺失、模型误判、外部环境变化。每类失败都要能映射到具体修复对象，避免泛泛地“优化 prompt”。

第二步 MVP 的成功标准：同类任务第二次、第三次执行时，能因为历史经验而减少步骤、减少失败、减少人工介入，并且没有增加安全风险或旧任务 regression。

两个 MVP 的闭环差异

第一步 MVP：可验证执行
User Task -> Workflow -> Tool / Computer-use -> Verifier -> Human Approval -> Trace Log

第二步 MVP：轻量自进化
Trace Log -> Diagnose -> Candidate Update -> Validate -> Approve / Merge -> Future Runs

5. 方向对比

方向解决的核心短板落地成熟度和 self-evolving 的关系

Long-horizon短任务强，长任务易崩中长任务自然需要经验复用和失败恢复

Bottom Line

Self-evolving agent 是下一代 agent 的“学习能力”，但下一代 agent 还需要手、眼、记忆、验证器、工具系统、协作机制和安全边界。
