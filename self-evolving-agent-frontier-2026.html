0. 一句话结论

2026 年论文多数仍很新，很多尚未经过长期复现和工程验证。这里把它们作为前沿方向信号，而不是把每个实验结论都当成稳定事实。
Core Takeaway

Self-evolving agent 的关键不是“会反思”，而是有明确的可修改对象、可追踪证据、独立验证、版本归档和安全边界。

2024-2026 的研究已经从早期的“让 LLM 反思并写一段记忆”，升级为“让 agent 系统的可变层被显式建模、被轨迹证据驱动、被独立 evaluator 验证、并在 archive、population、harness 中持续搜索”。

1. 哪些部分可以 Self-Evolve

最前沿的方向不是单点优化 system prompt，而是把 agent 拆成一组可进化对象。越靠近 harness、代码和参数，能力上限越高，安全和治理要求也越高。

1

Policy / System Prompt

任务策略、推理准则、rubric、meta-prompt。

2

Memory / Experience

原始轨迹、反思、经验树、失败模式、可检索案例。

3

Skills

可复用过程、代码片段、工作流模板、操作例程。

4

Tools

工具包装、组合、选择、接口修复和 retirement。

5

Workflow / Program

节点、边、调用顺序、分支、review loop。

6

Harness

执行环境、上下文、工具接口、生命周期、观测、治理。

7

Evaluator / Reward

自动评审、测试、critic、benchmark、virtual task。

8

Population

角色分工、团队选择、知识流、fork、merge、prune。

9

Code / Architecture

agent 自己的代码、编辑工具、上下文窗口策略、review 机制。

10

Fast Weights

LoRA、online adaptation、parametric memory，仍处早期。

3. 一个可用的总框架

Self-evolving agent 至少有三层循环。B 层是自进化核心，但不能脱离 A 层经验和 C 层治理。

A. Task-solving loop

user / task -> plan -> act / tool -> observe -> produce output

B. Learning / evolution loop

trajectory -> diagnose -> propose edit -> validate -> archive -> deploy / retrieve

C. Governance loop

sandbox -> permissions -> audit log -> regression suite -> safety reviewer -> rollback

1. Evidence Capture保存轨迹、工具调用、环境观察、错误、成本和反馈。
2. Failure Attribution判断失败来自 prompt、skill、tool、workflow、harness、memory 或 model。
3. Typed Proposal只在明确类型空间里生成候选修改。
4. Validation在 probe、held-out、regression、安全集上验证。
5. Archive / Deploy保留成功变体和失败证据，决定上线、灰度或丢弃。

4. 各部分目前最佳进化范式

4.1 System Prompt / Policy / Rubric

进化对象：system prompt、developer instruction、任务准则、输出格式、打分 rubric、风险规则、meta-prompt。

2026 更优范式：结构化、可归因、可审计的 policy edit。不要让 agent 自由改整段 system prompt，而是拆成 role constraints、allowed actions、decision rules、tool-selection rules、error-handling rules、verification checklist、safety rules 和 output contract。

最佳实践：离线可用 OPRO / PromptBreeder 做 prompt discovery；线上部署应转成结构化 policy diff。system prompt 分稳定宪法层、任务策略层、可进化经验层，只有后两层允许自动修改。

风险：semantic drift、删除关键约束、引入隐性目标、benchmark 过拟合。

4.2 Memory / Experience

进化对象：原始 episode、反思、错误模式、经验树、workflow trace、成功案例、失败案例、anti-pattern、检索索引。

2026 更优范式：raw episodes first + gated consolidation + structured experience management。

raw trajectory store   不可覆盖证据
case index             可检索案例
reflection notes       可丢弃/可替换
consolidated lessons   受控生成，需要验证
anti-pattern bank      失败签名 + 原因 + remedy
experience graph/tree  组织适用范围、前置条件、冲突关系

关键提醒：持续总结不一定让 agent 更聪明，可能让记忆越来越假。原始轨迹必须是一等公民。

4.3 Skills

进化对象：可执行代码技能、操作流程、任务模板、浏览器/GUI 例程、问题求解策略、领域方法。

2026 最佳范式：skill 不只是存储，而是经历 create / compose / validate / retire / specialize / generalize。

成熟 skill 应包含 name、intent、preconditions、inputs / outputs、procedure or code、required tools、validation probes、known failure modes、examples、version、usage stats、conflicts / complements、retirement criteria。

风险：skill 会互相冲突；单独有效不代表组合有效。

4.4 Tools

进化对象：工具选择策略、工具 wrapper、API 参数、错误恢复、工具组合、工具拆分、工具 retirement。

最佳范式：skill-tool atomic bundle。一次进化提案可同时包含新 skill、tool wrapper、validation probes、与现有 skills 的交互预期和 anti-pattern veto。

最佳实践：工具返回结构化；每个 wrapper 有 contract tests；高风险 tool policy 由 harness enforce。

4.5 Workflow / Agentic Program

进化对象：agentic workflow 的节点、边、分支、循环、reviewer、critic、tool-call 顺序、并行结构。

最佳范式：code-represented workflow + search + execution feedback。代码表示可 diff、可测试、可执行、可组合，也可被 agent 修改。

风险：workflow 搜索容易 benchmark overfit，必须做 held-out suite 与 cross-domain transfer。

4.6 Harness

进化对象：执行环境、工具接口、上下文装配、生命周期编排、observability、verification、governance、sandbox、权限、日志。

最佳范式：trace-guided harness diagnosis + scoped repair。

raw trajectory + harness code
  -> Harness-aware Trace IR
  -> step-level provenance + control-flow relation
  -> failure attribution to harness layer
  -> flaw record
  -> scoped repair operator
  -> regression validation

关键提醒：harness 的目标不是让 agent 更自由，而是让失败更可诊断、能力更稳定、安全边界更清楚。

4.7 Evaluator / Verifier / Reward

进化对象：测试集、probe task、自动评审器、reward model、critic prompt、rubric、milestone decomposition、安全评估。

最佳范式：multi-signal, evidence-grounded evaluation。

unit / integration tests
benchmark score
trajectory-level verifier
milestone-level evidence check
LLM judge with rubric
human or human-like oversight
cost / latency / robustness metrics
safety / policy checks
regression suite

风险：reward hacking。越是自进化系统，越要把 evaluator 当攻击面。

4.8 Multi-agent Population

进化对象：角色、团队组成、协作结构、知识流、agent 生命周期、specialization、fork/merge/prune/seed。

最佳范式：population archive + specialization-preserving knowledge transfer。

who collaborates
how they collaborate
how knowledge flows

知识传递应不对称、按 niche 路由，而不是全员共享 raw logs。

4.9 Code / Architecture

进化对象：agent 源码、编辑工具、上下文窗口管理、review 机制、工具调用代码、搜索算法。

最佳范式：open-ended archive + empirical validation + sandboxed self-modification。

sample existing agent from archive
  -> foundation model proposes code modification
  -> run benchmark / tests / safety checks
  -> add interesting high-quality variant to archive
  -> continue branching search

风险：这是最接近“agent 改自己”的路线，能力提升和安全风险同时最高。

4.10 Model Parameters / Fast Weights

进化对象：临时 LoRA、fast weights、parametric memory、online adaptation policy。

最佳范式：explicit memory + fast parametric adaptation 的双轨制。

实践判断：产品系统短期仍优先 non-parametric evolution：prompt、memory、skill、workflow、harness。参数自适应更适合高重复、可验证、可隔离的场景。

5. 不同自进化部分如何做评测

Short Answer

必须分开评测，也必须一起评测。分层评测负责 attribution，端到端评测负责 utility，长期评测负责 stability。

只做端到端评测不够，因为不知道到底是哪一层带来提升或退化；只做分层评测也不够，因为局部收益经常不转化成真实任务收益。稳妥做法是从单组件、相邻层交互、端到端任务、长期多轮、生产监控五级递进。

Level 1Component eval
单独测一个可进化对象是否变好。
Level 2Interaction eval
测它和相邻层组合后是否变好。
Level 3End-to-end eval
测真实任务成功率、成本、鲁棒性。
Level 4Longitudinal eval
测多轮 evolution 后是否持续变好。
Level 5Production eval
测线上漂移、失败级联、安全和人工介入。

5.3 统一实验设计

必要对照组

No-evolution、reflection-only、single-layer evolution、multi-layer evolution、oracle/human-designed upper bound、budget-matched baseline、held-out transfer、regression suite。

报告指标

success / quality gain、cost and latency、regression rate、safety drift、failure mode shift、component attribution、跨任务/跨域/跨模型迁移、plateau 轮数、人工介入。

Self-evolving 特殊曲线

performance over rounds、generalization gap、regression curve、cost per improvement、memory utility over time、archive health、safety drift。

最小判断标准

单层能证明边际收益；组合后能证明端到端收益；多轮后不显著退化；held-out 仍提升；成本没有失控；安全和旧能力没有被破坏。

Practical Recipe
  1. 先冻结 model 和 harness，只开放一个进化层，跑 3-5 轮。
  2. 每轮保存完整 event log：输入、输出、工具调用、状态、错误、修改 diff、验证结果。
  3. 每轮跑 evolution set、held-out set、regression/safety set。
  4. 每个候选修改必须有 component-level tests。
  5. 每轮做 ablation：新 memory/skill/workflow 打开和关闭各跑一次。
  6. 最后再开放多层共同进化，观察协同收益或互相破坏。

进化信号来自哪里

环境反馈轨迹证据LLM critic人类反馈群体反馈

最佳系统会混合信号并做 attribution，而不是把最终分数直接塞给 prompt optimizer。

如何避免越进化越差

保留 raw episode、held-out validation、regression suite、bounded edit space、rollback + versioning。

如何避免 reward hacking

proposer 与 verifier 分离；evaluator 有校准集；多 evaluator 交叉验证；隐藏集或轮换集；保留人工 spot check。

如何评估 self-evolving

看 learning curve、held-out transfer、compute-normalized improvement、regression rate、skill reuse、memory utility、safety drift、human intervention rate。

8. 研究前沿趋势判断

从 prompt evolution 到 system evolution

2023 的主角是 prompt/reflection；2024-2026 的主角变成 agentic system design、workflow code、harness、archive、population。

从单 agent memory 到 population memory

multi-agent 系统可以进化团队结构、知识流和 specialization，这是单 agent 不具备的新能力。

从自由文本总结到结构化经验管理

未来的 agent memory 更像数据库、版本系统、证据图，而不是聊天记录摘要。

Evaluator 成为瓶颈和攻击面

谁来判断“变好了”决定了 evolution 的上限，也决定了 reward hacking 的入口。

Harness 会成为产品护城河

同一个模型，换不同 harness，能力差距会很大。好的 harness 可诊断、可回放、可验证、可治理。

开放但有边界

最可能落地的是不可变 core + 可进化插件层 + 审计日志 + 自动验证 + 人类审批。

10. 定义与 Checklist

English

A self-evolving agent is an agentic system that uses evidence from its own or peers' trajectories to propose, validate, archive, and deploy bounded modifications to one or more explicit system components, such that future behavior improves under held-out evaluation without requiring manual redesign of those components.

中文

Self-evolving agent 是一种能从自身或群体运行轨迹中提取证据，对明确的系统组件提出有边界的修改，并通过验证、归档、部署来改变未来行为的 agent 系统；它的关键不是“会反思”，而是“有可修改对象、可验证信号、可回滚版本和长期复用机制”。

最简 Checklist

如果这些问题答不上来，它更可能只是 reflective agent，不是成熟的 self-evolving agent。

1它到底改了什么？prompt、memory、skill、tool、workflow、harness、evaluator、代码、参数，还是只是重新生成答案？
2修改依据是什么？最终分数、轨迹证据、错误归因、人工反馈、peer history？
3修改空间是否有类型和边界？
4是否保留原始轨迹，而不是只保留总结？
5是否有独立 verifier / held-out eval / regression suite？
6是否有 archive，而不是只保留当前最好版本？
7是否能跨任务迁移，而不是只修某一道题？
8是否记录失败模式和反例？
9是否能回滚？
10是否有安全边界和人工介入点？
